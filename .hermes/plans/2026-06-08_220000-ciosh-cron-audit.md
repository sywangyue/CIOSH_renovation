# CIOSH Intel Radar Cron 配置审计 & 迁移方案

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** 确保 CIOSH 情报雷达日报（北京 03:00）和周报（北京周一 03:30）自动触发，用户无需手动管理 cron 时间。

**Architecture:** 从 macOS 系统 crontab（UTC 时区）迁移到 Hermes cronjob（本地时区），使用 `no_agent=true` + `script` 模式直接执行 shell wrapper。

**Tech Stack:** Hermes cronjob, bash, Python 3, macOS

### 核心问题

1. **时区混乱**：macOS cron 使用 UTC，`0 19 * * *` = 北京 03:00。用户（及 CLAUDE.md）用北京时间思考，crontab 用 UTC 表达，心智负担大。
2. **"手动写入"问题**：当前 crontab 是通过 `setup_cron.sh` 一次性写入的，时间硬编码在脚本和 crontab 两处。用户要求"不手动写入 cron job 时间"。
3. **无 venv**：依赖挂在全局 python3，环境不可复现。
4. **从未成功运行**：日志目录为空，cron 可能因权限/路径/外置盘未挂载等问题静默失败。

## 方案选择

### 方案 A：迁移到 Hermes cronjob（推荐）

将 CIOSH 的日报/周报从系统 crontab 迁移到 Hermes cronjob，使用 `no_agent=true` + `script` 模式。

**优点：**
- 时间使用本地时区（北京），`0 3 * * *` 直接表达"凌晨3点"，无需 UTC 换算
- 用户通过 `hermes cronjob` 命令管理，不碰 crontab
- Hermes 已在运行且管理其他 cron 任务，统一调度
- `no_agent=true` 模式零 token 消耗，只执行脚本

**缺点：**
- Hermes 进程必须运行中（当前已运行）
- 外置盘未挂载时同样失败（与系统 cron 一致）

### 方案 B：保持系统 crontab，修复问题

修复 venv、日志等问题，保持当前架构。

**缺点：**
- 仍然需要手动管理 crontab（与用户要求冲突）
- UTC 时区继续造成混淆
- macOS cron 在睡眠时不执行，Hermes 同样有此限制

### 推荐：方案 A

### Task 2: 创建 Hermes cronjob — 日报

**Objective:** 创建日报 cronjob，北京时间每天 03:00 执行。

**Step 1:** 创建 cronjob
```
hermes cronjob create \
  --name "CIOSH 日报" \
  --schedule "0 3 * * *" \
  --script "/Volumes/databoard/AI Project/ciosh/intel/scripts/run_daily.sh" \
  --no_agent \
  --deliver local \
  --profile default
```

参数说明：
- `schedule "0 3 * * *"`：北京每天 03:00（Hermes 使用本地时区）
- `script`：指向 shell wrapper（含外置盘等待逻辑）
- `no_agent`：不启动 LLM，直接执行脚本
- `deliver local`：stdout 保存到本地，不推送消息（脚本自己发邮件）

**Step 2:** 验证 cronjob 已创建
```bash
hermes cronjob list | grep -A5 "CIOSH 日报"
```

## 风险

| 风险 | 缓解 |
|------|------|
| Hermes 进程未运行 | `hermes cronjob` 守护进程独立于 CLI session，检查 `hermes status` |
| 外置盘未挂载 | shell wrapper 已有 30s 等待逻辑；如果盘彻底不在，日志会记录 ERROR |
| API key 过期 | .env 保持不变，本次迁移不涉及 |
| Hermes 升级影响 cronjob | Hermes cronjob 格式向后兼容，不受影响 |
