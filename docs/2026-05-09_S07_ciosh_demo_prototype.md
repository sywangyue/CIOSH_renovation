# CIOSH AI 展会助手 Demo — 原型方案

**日期**：2026-05-09
**类型**：Demo 原型计划
**目标**：面向 CIOSH 项目组演示的假前端 + DeepSeek API 对话界面
**用途**：讲清楚"这是什么"，决定是否进入正式开发

---

## 一、目标与定位

### 这不是开发项目，这是"说服工具"

| 维度 | 说明 |
|------|------|
| 受众 | CIOSH 项目组（Rya、Kate、CC顾问） |
| 目的 | 让他们看到"AI 展会助手长什么样、能做什么" |
| 效果 | 在微信/浏览器里打开 → 跟 AI 聊展会 → 立刻理解价值 |
| 不做的事 | 不用后端、不用数据库、不用登录、不用真实展商数据 |
| 时间 | 1-2 天可完成 |

### Demo 要回答的三个问题

1. "这玩意儿是什么？" → 一个聊天窗口，像客服但更聪明
2. "它能做什么？" → 查展商、找产品、问展会信息
3. "为什么要做？" → 观众/买家不需要翻目录，直接问 AI

---

## 二、技术方案

### 最简实现

```
单个 HTML 文件
  ├── Chat UI（模仿微信/客服对话风格）
  ├── 调用 DeepSeek API（fetch + SSE 流式输出）
  └── 系统提示词预设 CIOSH 行业知识
```

**不需要**：
- 不需要服务器（静态 HTML 即可）
- 不需要后端（DeepSeek API 直连）
- 不需要数据库（假数据写在提示词里）
- 不需要域名（本地打开或 GitHub Pages / Vercel 免费部署）

**唯一依赖**：DeepSeek API key（已有）

### 部署方式

| 方式 | 做法 | 适用场景 |
|------|------|---------|
| 本地 HTML | 双击打开 | 你自己演示 |
| GitHub Pages | push 到 repo | 发给项目组链接 |
| Vercel | 一键部署 | 同上，更快 |

### 3.3 设计原则（来自官网观察）

1. **绿色即品牌**：所有可点击、可强调的元素统一使用 `#009040`
2. **白底为主**：大面积白色背景，绿色做功能区隔
3. **字体克制**：只用一个字体栈，通过字号和字重区分层级
4. **橙色为点缀**：仅用于特殊强调（价格、重要日期、行动号召），不可滥用
5. **不创造任何设计语言**：以上全部基于 Branding 文件 + 官网实测

### 3.3 交互细节

- **流式输出**：AI 回复逐字显示（打字机效果），体现"AI 正在思考"
- **卡片展示**：展商信息用卡片承载，含关键字段
- **快捷按钮**：底部固定 3-4 个常用问题，点击即发送
- **加载状态**：发送后显示"..."跳动动画
- **移动端适配**：优先模拟微信内置浏览器宽度（375-414px）

> "这是 CIOSH 展会 AI 助手的原型。观众/买家打开这个链接就能用。"

### 第二幕：查展商（30秒）

> 输入："帮我找南通做手套的厂"
> → AI 返回卡片式结果

### 第三幕：问展会信息（20秒）

> 输入："展会什么时候在哪"
> → AI 回答时间地点交通

### 第四幕：行业知识（30秒）

> 输入："EN388是什么"
> → AI 解释标准 + 推荐相关展商

### 第五幕：价值阐述（30秒）

> "这只是一个 demo，但核心逻辑已经跑通了。正式版可以：
> 1. 接入真实展商数据库
> 2. 展商可以上传自己的产品
> 3. 买家提问自动匹配展商
> 4. 对接到微信群里直接用
> 5. 主动去小红书/抖音帮展商找买家"

## 八、难度评估

| 维度 | 评估 |
|------|------|
| 技术难度 | ★☆☆☆☆ 极低 — 纯前端 HTML + API 调用 |
| UI 设计难度 | ★★☆☆☆ 低 — 参考微信/客服聊天界面 |
| 时间投入 | 半天到一天 |
| API 成本 | 几乎为 0（demo 用 DeepSeek，几句对话几毛钱） |
| 风险 | 无 |

**结论：零技术风险。唯一的投入是你的时间。**

## 九、不做的事（避免 over-engineering）

- ❌ 不做后端
- ❌ 不做数据库
- ❌ 不做用户系统
- ❌ 不做真实数据对接
- ❌ 不做复杂前端框架
- ❌ 不做部署自动化

**Demo 的唯一使命：让项目组说"明白了，继续做"。**
