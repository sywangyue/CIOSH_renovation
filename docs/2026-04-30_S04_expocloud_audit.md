### 1.7 已知问题 / 异常记录

1. **展会已结束**：展会状态显示「结束」（2026-04-09结束），部分模块数据为历史快照，不可再操作。
2. **财务数据不一致**：首页展位信息卡显示「总金额 7.9w / 已付 0」，参展信息页显示「总金额 ¥97,600 / 已付 ¥0」，收款记录中有一笔 ¥79,040 已付（付款方：杜塞尔多夫展览上海），差异原因推测为代理商代付逻辑与系统计算口径不同。
3. **展商秀未开通**：ExhibitorShow.IsEnabled=false，展商端无法体验展商秀相关功能，需主办端开通后才可访问展商展示模块。
4. **主办端权限局限**：展商端仅能访问自身邀请的289条观众数据，全量观众数据（1,181点击 / 287完成注册）需主办端账号补充调研。
5. **数据维护无记录**：数据维护任务列表为空，无法确认导入/导出任务的完整操作流程。
6. **企查查组件**：页面加载了企查查 JS 组件，推测用于企业信息自动填充，但展商端使用场景不明（可能仅主办端使用）。

| 核心定位         | 展商营销展示平台（展品/直播/H5展台） | 展会运营管理平台（合同/付款/观众邀请）                            |
| 主要用户场景       | 展商对外展示（展品/活动/品牌）     | 展商对主办方提交资料+邀请观众                                 |
| 数据流向         | 展商→观众（单向展示）          | 展商↔主办（双向），展商→观众→回流数据                            |
| Feature Flag | 无（固定功能集）             | **有**（模块化开关，Bpm/Bmm/ExhibitorShow等）             |
| 多SaaS集成      | 独立系统（无集成迹象）          | **已集成展商秀入口**（ExhibitorShow.IsEnabled=false，可开通） |


> **重大发现**：慕渊会展云（SaaS-B）已内置展商秀（SaaS-A）作为可选功能模块（ExhibitorShow），说明两个平台在底层已有集成协议，并非完全独立系统。

## ── 第三部分：合并策略分析（顾问区）──

### 3.1 合并目标重新定义

基于双平台调研，将「合并两个SaaS到小程序」拆解为两个子目标：

**目标A（展会运营流）**：替代 SaaS-B（慕渊会展云）的展商服务流程

- 参展信息/合同/发票的移动端查看
- 楣板确认/会刊提交的移动端操作
- 观众邀请（海报生成/分享/邮件发送）的小程序原生实现

**目标B（展商营销流）**：替代 SaaS-A（展商秀）的展商品牌展示功能

- 展品展示/公司介绍的小程序页面
- H5 展台迁移为小程序展台
- 活动申请/直播间的小程序封装

### 3.2 可直接小程序化的功能（优先级 P0）


| 功能             | 来源SaaS | 技术路径                  | 数据接口                                    |
| -------------- | ------ | --------------------- | --------------------------------------- |
| 观众邀请海报生成       | SaaS-B | 小程序画布API生成海报，含二维码     | v-api.expo2345.com/graphql（已有）          |
| 邀请统计 Dashboard | SaaS-B | 小程序首页卡片               | trackTotal.exhibitorVipTrackInviteTotal |
| 我的观众列表         | SaaS-B | 小程序列表页+搜索             | visitor.page（已有完整schema）                |
| 楣板确认/企业信息确认    | SaaS-B | 小程序表单页                | exhibitor.view mutation                 |
| 展位平面图          | SaaS-B | 小程序 canvas 或 web-view | 需坐标数据接口                                 |
| 展品展示           | SaaS-A | 小程序列表+详情页             | 需 SaaS-A GraphQL                        |


### 3.3 合并的技术前提

1. **认证统一**：SaaS-B 已使用标准 OIDC，小程序可通过微信授权换取 SaaS-B 的 Bearer Token（需主办方开放微信小程序 AppID 绑定）
2. **API 可访问性**：SaaS-B 的 `api.expo2345.com/graphql` 和 `v-api.expo2345.com/graphql` 均有 CORS 策略（允许 `exhibitor.expo2345.com` 跨域），小程序需要主办方将小程序域名加入白名单
3. **展商秀集成**：SaaS-B 已有 ExhibitorShow.HostId 字段，说明展商秀可通过主办方配置接入，无需独立API对接

### 3.4 不可迁移 / 暂缓迁移的功能


| 功能          | 原因                       |
| ----------- | ------------------------ |
| 直播间         | 需要 SaaS-A 主办端支持，短期无替代方案  |
| 证件申请        | 线下制证流程，依赖主办方后台处理         |
| 贸易洽谈/询盘     | 复杂商务流程，非核心路径             |
| 数据导出（Excel） | 小程序端无文件系统，可降级为分享/发送到邮箱   |
| 会刊信息提交      | 提交后需人工审核，截止日期管控，暂保留PC端处理 |


### 3.5 建议的 POC 优先级（小程序阶段）

```
Sprint 1（核心数据流）
  ├── 微信登录 → 换取 SaaS-B OIDC Token
  ├── 首页 Dashboard（展位信息 + 邀请统计）
  ├── 海报邀请（生成 + 分享）
  └── 我的邀请列表（观众名单查看）

Sprint 2（运营流程）
  ├── 参展信息查看（订单/收款/发票）
  ├── 楣板确认 / 企业信息确认（表单提交）
  └── 展位平面图（只读展示）

Sprint 3（营销展示）
  ├── 展品管理（接入 SaaS-A 或重建）
  ├── 公司信息展示页（to-B 名片）
  └── 活动信息展示（接入 SaaS-A 或重建）
```

### 3.6 关键风险

1. **API 授权壁垒**：两个 SaaS 的 API 均有自定义签名头（x-adprv-sign），小程序直接调用需主办方开放白名单或提供 SDK。
2. **观众实名数据合规**：v-api 返回完整身份证号，小程序端展示需符合数据合规要求，建议脱敏处理。
3. **展商秀开通条件**：ExhibitorShow.IsEnabled 由主办端控制，需确认本届CIOSH是否计划开通，否则展商展示功能需从 SaaS-A 单独对接。
4. **跨平台数据一致性**：SaaS-A 和 SaaS-B 的展商数据（公司名/展位号）来自不同数据库，合并后需确认主数据源。
